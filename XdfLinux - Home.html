XdfLinux is formed by a bootsector and a small bootprogram that can boot a Linux kernel and an initial ramdisk from a Xdf formatted floppy disk.

The program is released under the GNU General Public License.

What is Xdf?

Normally floppies are formatted to contain 1440KB of data: 80 tracks, 2 sides and 18 512-byte sectors. The layout of a single track is shown in the figure below. Note that the right and left sides of the figure are connected in reality.
Linux formats those floppies with a skew of 2 sectors for head changes and 5 for changing tracks: 2 7/18 rotations per track. A floppy with this format can be read in about 38 seconds (38KB/s).

It is clear from the figure that a lot of space is wasted between the end of data of a sector and the start of the header of the next sector. In fact it is possible to format up to 21 sectors per track. This way one can format disks with capacities of 1520KB, 1600KB, and 1680KB. The 1680KB format does have a problem, however: adjacent sectors come too close together to be read consecutively. The usual solution is to interleave them, which makes the effective transferrate much lower.
This floppy format can be read in about 67 seconds (25KB/s).

The Xdf floppy format makes use of the fact that for some odd reason the floppy drive controllers allows formatting sectors with the "wrong" headers. The 1840KB floppies formatted with xdfcopy use 128-byte sectors of which selected headers indicate a 512-byte sector, a 1KB-sector, a 2KB-sector and a 8KB-sector on each side. When these sectors are written to, the remaining 128-byte sectorheaders are overwritten by the data.
All of the data in the track can be read in 2.5 rotations, which involves switching sides a few times, in contrast to normal formats which are read one side at a time. This trick makes it a format with a really high effective transferrate: reading a whole disk takes only 40 seconds (46KB/s). For compatibility reasons, track 0 of a Xdf floppy is formatted with 512-byte sectors (19 per side). The bios can thus locate a suitable bootsector, and ignorant programs will see a tiny fat filesystem.

Note that this trick of switching sides after a sector could be used to implement a faster 21-sector format:
If this is read in the order 1top, 1bottom, 13top, 13bottom, ... 12top, 12bottom, ..., 2top, 2bottom, ..., 11top, 11bottom. Reading the track in the normal order takes four full revolutions, while the better order needs only three full revolutions. This all assumes that your floppy controller can switch sides fast enough, but if it cannot, that can be detected and reading/writing can than be done in the normal order.

Finally, it is possible to cram even more on a floppy, using only two sectors: one sector of 4KB and one of 8KB. This gets very close to the limit of the raw capacity of the drives, so drives that are a bit out of spec may not work all right. Track 0 of a Xdf floppy is formatted with 512-byte sectors (20 per side, but the last sector on the second head is not used). If it works you will get 1920KB of capacity, readable in about 40 seconds (48KB/s).

What is the on-disk format?

The Xdf floppies contain a tiny fat filesystem of eight 512-byte sectors on the first track of the floppy. The filesystem consists of the bootsector, two fats and a single-sector root-directory. Four sectors are left for files. The Xdf filesystem has its own bootsector, fat, and rootdirectory, all on the first cylinder of the floppy, but the sector numbering scheme is made inaccessible by using sector numbers in the range 129 (0x81) and up. Note that there is only one fat on disk, while the bios parameter block in the Xdf bootsector informs that there are two copies. A few sectors of the data area of the Xdf filesystem are still on the first cylinder, but the greater part is located on the generic tracks.

Download / howto use it

If you want to try the program, you need to download xdflinux.ash, xdflinux.asm and xdfulilo.asm (Current version, July 22, 2003, see readme/changelog file). You will also need NASM, Fdutils, and Mtools, and a Linux kernel (bzImage format) and an initrd file, together no more than 1816KB (Xdf) or 1895KB (XXdf), to see XdfLinux in action.

I use the following commands to write them to a floppy:
nasm -DXDF=1840 xdflinux.asm
nasm -DXDF=1840 xdfulilo.asm
xdfformat -1 -D x: /dev/fd0
mdeltree a:
fdrawcmd track=0 length=512 write 0 0 0 1 2 1 0x1b 0xff < xdflinux
mcopy xdfulilo a:XDFULILO
mcopy kernel a:LINUX
mcopy initrd a:INITRD
For the 1920KB format change -DXDF=1840 to -DXDF=1920 on the command lines of nasm, and change -1 to -3 on the xdfformat command line. Note that "x:" points to an entry in my mtools.conf file. To be able to read the xdfformatted floppy and its stub using mtools, I changed the mtools.conf file as follows:
# # Linux floppy drives
drive a: file="/dev/fd0" exclusive use_xdf=1
drive x: file="/dev/fd0" exclusive
Note that XdfLinux is still very preliminary software. It does not understand the structure of the filesystem. The bootloader only works if the files are written in the order XDFULILO, LINUX, INITRD, starting from an empty filesystem! XDFULILO has to be exacty 2048 bytes long. The only information gathered from the filesystem are the lengths of the files registered at the second and third directory entries of the root directory. The names of the files are not used at all. As long as they are in UPPERCASE MSDOS 8.3 format the names used do not matter at all.

If you use this bootloader, I would very much appreciate a note (heukelum@phys.uu.nl). If you give an URL (preferably with a short description of the site where it points to), I'ld be happy to add it to the links section below.

Using this software is at your own risk, of course!

P.S. The first report of computers not compatible with XdfLinux has come in. Both are HP Vectra machines. One "HP Vectra-733" aborts reliably with a read error as soon as it starts to read the kernel from the generic tracks. A "HP Vectra VL" gives read errors on many occasions and, at any rate, reads the generic tracks slowly. At the moment I still categorize those failures as "BBS" (broken bios syndrome). If you get XdfLinux to work on some machines, but not on others, I'ld like to hear about that too...

Interesting links for making small linux distributions:
Busybox (the Swiss Army Knife of Embedded Linux)
A highly configurable multi-call binary. It implements a lot of the standard linux tools, but it is much smaller than the separate tools.
uClibc (i.e., μClibc -- a C library for embedded systems)
A C-library that aims to be as compatible as possible with glibc, but smaller.
UPX (the Ultimate Packer for eXecutables)
A program that can compress executables, without losing the ability to execute them. The development version can recompress linux kernels: we are waiting for version 2.0! UPX-NRV is binary-only freeware that uses the NRV library for compression. UPX-UCL is open source, but compression is a bit less efficient.
UCL (the UCL Compression Library)
An open source library of compression/decompression algorithms. Decompression is very fast and in-place. Compression is a bit slow, but speed/compression ratio are tunable. If compression speed is not an issue, it compresses much better than gzip -9.
bz2bzImage (bzip2 compressed kernels and/or initial ramdisks)
A patch that adds bunzip2 capability to current 2.4.xx kernels. With this patch you can make bzip2-compressed kernels and/or add support for bzip2-compressed initial ramdisks.
